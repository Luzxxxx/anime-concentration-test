# 二次元浓度测试稳定化改造设计

日期：2026-07-15

## 1. 背景

现有作品已经具备首页、模式选择、答题反馈、结果分析、历史、成就和分享通行证等完整体验骨架，视觉主题为“樱花学园通行证”。当前主要风险不在功能数量，而在测评可信度和若干未收口状态：答案位置严重偏斜、理论满分计算错误、分类样本过少却输出强结论、首页按钮无交互、二次答题残留上一局分数、分享卡分母和二维码错误，以及结果页过长、分享入口过晚。

本次改造在原项目中完成，不复制或重建项目。实施时在 TRAE 中为现有项目新建独立 AI 对话，使用 `/plan` 管理步骤；涉及方向性判断时可使用 `/brainstorming`，但所有代码修改必须遵循“单项修改、立即验证、通过后继续”的节奏。

## 2. 目标

1. 让知识段位能够真实反映不同模式下的答题表现。
2. 消除通过固定答案位置猜题的捷径。
3. 保证能力画像具有最小分类样本量，并诚实表达数据不足。
4. 修复比赛现场可见的死按钮、状态残留、空区域和分享错误。
5. 保留樱花通行证整体视觉，同时压缩首页空白和结果页信息长度。
6. 建立可重复运行的题库、计分和浏览器回归检查。

## 3. 非目标

- 不整体更换前端框架。
- 不重画全部视觉资产。
- 不在稳定化阶段继续增加题目、成就或复杂社交系统。
- 不引入账号体系、云端数据库或公开排行榜。
- 不用大规模重构替代明确缺陷修复。

## 4. 总体架构

保留 `index.html` 作为交付入口和现有页面结构。仅将高风险、需要独立验证的逻辑划分为清晰边界：

- 题库校验：检查结构、重复、分类、难度和题型覆盖。
- 题目会话生成：抽题、分类配额、难度配额和选项随机化。
- 计分与段位：知识成绩、竞技得分、理论上限和段位映射。
- 结果画像：分类样本量、能力标签和数据不足状态。
- 分享：分享文案、海报数据、公开地址和二维码。
- UI 状态：页面切换、答题重置、历史、成就空态和反馈。

这些边界可以继续以内联模块存在，也可以按 TRAE 的实施计划拆为少量脚本文件；拆分不是目标，能独立测试才是目标。

## 5. 第一轮：可信度与明确缺陷

### 5.1 题目会话与选项随机化

每次开始测试时从原始题库创建题目副本。四个选项使用 Fisher-Yates 洗牌，并根据原正确选项内容重新计算答案索引。原始题库对象保持只读，历史记录只保存会话结果，不保存对题库对象的引用。

分类配额为：轻松模式每类至少 2 题，标准模式每类至少 3 题，挑战模式每类至少 4 题。抽取算法先满足分类配额，再在各分类内部满足整体难度比例；题库不足时允许降低难度配额，但不得静默降低分类覆盖。若任何分类最终少于 2 题，结果页对该分类显示“样本不足”，不生成“达人”或 100% 能力结论。

### 5.2 知识段位与竞技得分分离

知识成绩仅使用难度加权正确率：

`知识成绩 = 答对题目的基础分总和 ÷ 当前题组基础分总和 × 100%`

基础分沿用简单 10、中等 15、困难 20。知识段位按照现有六段区间映射，但输入固定限制为 0–100%。

竞技得分用于娱乐和同模式比较：

`竞技得分 = 基础分 + 速度奖励 + 每题对应的连击奖励`

理论竞技满分必须逐题模拟全对、速度奖励和连击奖励，不能只在末尾增加一次连击上限。不同模式不比较原始竞技得分；如显示历史百分位，只比较同模式下的归一化竞技比例。

### 5.3 题库质量检查

增加可重复运行的检查，至少覆盖：

- ID 唯一。
- 题目文本和选项字段完整。
- 每题恰好四个选项，答案索引合法。
- 重复题目和高度相似题目清单。
- 分类、难度、题型数量统计。
- 语音题必须包含可播放文本。
- README 与实际题库数量、题型一致。

事实准确性无法完全自动判断。已发现的疑似错误和歧义进入人工复核清单，逐题记录权威来源与修正结论。未完成全库复核前，比赛版本至少修复已经命中的错误、重复题和解释矛盾。

### 5.4 UI 功能收口

- “查看历史”滚动并聚焦历史区域。
- “选择模式”进入模式页并渲染当前选择。
- 新一轮答题前将可见分数、连击、进度条、反馈和按钮文案全部重置。
- 历史卡显示日期、模式、知识成绩、正确数和段位。
- 无新成就时隐藏整个“本次解锁成就”容器。
- 非选中的模式不向辅助技术朗读“已选”。
- 隐藏的成就弹窗设置正确的 `aria-hidden` 或 `inert` 状态；打开时转移焦点，关闭时恢复焦点。

### 5.5 分享可信度

分享文案与海报统一使用以下字段：模式、知识成绩、竞技得分、正确数/总题数、段位和最强/待提升分类。

不再把题目总数写成竞技满分。只有同模式历史不少于 5 条时才显示“超过本机同模式历史 X%”；否则显示“本机第 N 次记录”或不显示比较数据。

新增可配置的公开作品地址。配置有效公开地址时生成可扫描二维码；地址缺失或仍为 localhost 时隐藏二维码，不绘制占位图案。原生分享取消、文件分享失败、剪贴板失败都必须显示明确但不打扰的反馈。

## 6. 第二轮：视觉与用户体验

### 6.1 首页

- 保留樱花通行证主题和开始测试主按钮。
- 开始按钮附近明确说明默认题量、预计耗时和测试产出。
- 历史、模式和成就整合为紧凑工具区，不再纵向散落。
- 显示一张可进入详情的最近成绩卡。
- 原创角色“小星”成为主要识别资产；官方作品海报降为次要点缀或题目线索。
- 首屏图片改为合适尺寸的 WebP/AVIF，并延迟加载非首屏资源。

### 6.2 答题页

- 增加明确的 `当前题号/总题数`。
- 轻松和标准模式弱化 30 分钟倒计时；挑战模式保留具有意义的限时表达。
- 保留即时正确/错误反馈、解释、连击和得分反馈。
- 返回操作继续要求确认，确认文案说明当前进度会丢失。

### 6.3 结果页

结果首屏只包含：段位、知识成绩、竞技得分、优势/待提升分类和“生成并分享通行证”主按钮。

能力雷达与三条个性化建议位于首屏之后。分类明细、错题手账、成就和历史使用折叠区，默认只展开最有价值的一项。删除重复分享入口，分享主动作保持单一。移动端没有新成就时不得出现空白卡片，关键分享动作无需滚动数屏才能到达。

## 7. 数据流

1. 用户选择模式。
2. 会话生成器按分类和难度抽取题目，并随机化选项。
3. 答题引擎记录选择、正确性、基础分、用时和连击。
4. 结果计算器分别生成知识成绩和竞技得分。
5. 画像模块根据各分类样本量输出百分比、数据不足状态和建议。
6. 存储模块保存版本化结果摘要。
7. 分享模块从同一结果摘要生成文案、海报和可选二维码。

结果摘要增加数据版本号，以便旧的 LocalStorage 数据在字段变化后安全迁移或忽略不兼容字段。

## 8. 错误处理

- LocalStorage 读取、解析或迁移失败时回退为空状态，并提供一次非阻塞提示。
- 题目会话无法满足分类配额时记录清晰错误并阻止开始，不生成误导性画像。
- 二维码配置无效时隐藏二维码区域，不输出假码。
- 分享被用户取消时不报错；真正失败时给出重试或下载图片选项。
- 语音不可用时显示明确的文字降级说明，并将题型称为“台词听辨”，不宣称声优识别。

## 9. 验收与测试

每项修改必须先完成定向检查，再继续下一项。

### 自动检查

- 全错时知识成绩为 0%，全对时为 100%。
- 不同模式使用各自题组基础分上限，段位不依赖原始分数大小。
- 理论竞技满分等于逐题全对且获得全部速度、连击奖励的合计。
- 多次有种子会话中，正确选项位置接近均匀且原答案语义保持正确。
- 10/15/20 题模式分别满足每类 2/3/4 题或明确失败。
- 题库结构检查无非法答案、重复 ID 和缺失必填字段。

### 浏览器回归

- 首页三个工具入口均有可见结果。
- 第一次和第二次开始答题时分数、连击均为零。
- 完成一轮测试后，知识段位、竞技得分、分类样本量和历史卡一致。
- 保存海报中的字段正确，二维码在配置公开地址时可扫描，未配置时不出现。
- 分享取消和失败具有正确反馈。
- 桌面端和 375px 移动端没有横向溢出，结果主操作在合理位置。
- 控制台无新增错误或警告。

## 10. TRAE 实施方式

在 TRAE 中打开现有 `traebisai` 项目并新建 AI 对话。先向新对话提供本规格文件，再使用 `/plan` 将工作拆成以下顺序：

1. 建立基线检查和题库校验。
2. 修复选项随机化、分类抽题和知识/竞技双成绩。
3. 修复首页入口和第二次答题状态。
4. 修复分享数据、百分位和二维码。
5. 修复历史、成就空态和无障碍状态。
6. 优化首页信息层级。
7. 优化结果页首屏和折叠详情。
8. 完成桌面、移动端和完整流程回归。

每个步骤要求 TRAE 先说明将修改的文件和验收方法，修改后立即运行相应检查。若检查失败，先修复当前步骤，不得继续批量实施后续步骤。
