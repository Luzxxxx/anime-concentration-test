# 二次元浓度研究所：GitHub 作品集发布设计

## 1. 背景与目标

项目当前是一套可运行的原生 HTML/CSS/JavaScript 单页应用，包含 226 道题、三种挑战模式、双计分、角色匹配、成就、历史记录、分享海报与 PWA 支持。现有实现主要集中在一个约 380 KB 的 `index.html` 中，能够演示，但不利于维护、测试和公开展示；仓库尚未初始化 Git，也缺少完整许可证文件、自动化质量检查和部署流程。

本轮目标是把项目升级为公开 GitHub 作品集：保留“二次元浓度研究所 / 樱花街机学院”品牌与核心玩法，适度精修 UI，引入 Vite 工程化、自动测试和 GitHub Actions，并部署可在线试玩的 GitHub Pages 站点。

## 2. 已确认的产品决策

- 仓库定位：公开作品集，面向招聘方、开发者和 ACGN 兴趣用户。
- 视觉方向：采用“平衡精修发布版”，保留糖果色、厚描边、硬边阴影和樱花街机学院主题，不进行彻底换肤。
- 工程方向：采用 Node.js + Vite，允许拆分现有单文件结构，并增加测试、代码检查和自动部署。
- 素材策略：保留现有官方公开图片；在 README、素材来源文档和许可证说明中明确版权归原权利方所有，MIT 仅适用于原创代码和原创素材。
- 发布方式：默认创建公开 GitHub 仓库，并通过 GitHub Actions 发布 GitHub Pages。

## 3. 方案比较与选择

### A. 平衡精修发布版（已选）

保留现有视觉辨识度，改善页面层级、响应式细节和交互反馈；同时完成 Vite、模块拆分、测试、文档、CI 和 Pages。该方案最适合作品集，能够同时展示产品完成度与工程能力，且不会破坏已有玩法。

### B. 沉浸式视觉重塑

改为深色赛博街机视觉并增加大量动效。视觉冲击更强，但偏离现有品牌，改动和回归风险较大，不适合本轮发布节奏。

### C. 工程优先简洁版

仅做必要的 UI 修复，把资源集中在代码拆分和自动化。交付风险最低，但对作品集第一印象的提升不足。

## 4. 信息架构与页面体验

应用继续采用单页多状态流程：

1. 首页：说明价值、显示模式与题量、提供开始测试和历史/成就入口。
2. 模式选择：对比轻松、标准和挑战模式，明确题量、预计时间和难度。
3. 答题页：顶部 HUD 展示进度、分数、连击和计时；主体展示题目媒体、选项和反馈。
4. 结果页：展示知识掌握度、竞技表现、能力雷达、段位、角色匹配和分享入口。
5. 弹窗层：历史、成就、分享、设置和危险操作确认保持统一交互。

UI 精修聚焦以下内容：

- 统一标题、正文、说明文字和数字的视觉层级。
- 压缩首页冗余信息，让主行动按钮更突出。
- 统一卡片描边、阴影、圆角、悬停、按下和焦点状态。
- 提升手机端模式卡、答题选项、结果数据块和弹窗的可读性。
- 加入轻量页面进入与结果揭晓动效，并尊重 `prefers-reduced-motion`。
- 保留现有官方图片拼贴风格，同时增加遮罩和排版约束，避免素材主导界面。
- 修正元数据、图标、PWA 名称、版本文本和分享卡片的一致性。

## 5. 工程架构

项目采用 Vite 驱动的原生 JavaScript 架构，不引入 React/Vue 等运行时框架，避免为当前规模增加不必要复杂度。

```text
traebisai/
├── index.html                 # 精简后的应用 HTML 骨架
├── public/
│   └── assets/                # 图片、PWA 图标与素材来源说明
├── src/
│   ├── main.js                # 应用启动与事件绑定
│   ├── styles/                # 设计令牌、基础、组件、页面、响应式样式
│   ├── data/                  # 题库、角色与成就数据
│   ├── core/                  # 计分、抽题、匹配、校验、存储等纯逻辑
│   └── ui/                    # 页面渲染、弹窗、分享与可访问性行为
├── tests/                     # Vitest 单元测试
├── scripts/                   # 发布前校验等辅助脚本
├── .github/workflows/         # CI 与 GitHub Pages 部署
├── README.md                  # 作品集式项目介绍
├── LICENSE                    # 仅覆盖原创代码与原创素材的 MIT 文本
└── THIRD_PARTY_ASSETS.md      # 第三方图片版权与来源边界
```

拆分遵循“先保持行为，再改善边界”的原则：题库、配置和纯逻辑独立为模块；DOM 密集的 UI 行为按页面或职责拆分；不在本轮重写全部业务逻辑。

## 6. 数据流与状态

- `main.js` 初始化存储、验证题库、读取用户设置并展示首页。
- 用户选择模式后，抽题模块根据模式配置与分类覆盖规则生成本轮题目。
- 每次作答由计分模块返回正确性、基础分、速度分和连击分，应用状态记录答案与耗时。
- 完成测试后，结果模块汇总知识掌握度、竞技表现、分类分数、段位和角色匹配。
- 历史、成就与设置通过版本化存储适配器写入 `localStorage`；损坏或旧版本数据自动回退到安全默认值。
- 分享模块只读取最终结果，生成海报和可复制摘要，不修改测验状态。

## 7. 错误处理与兼容性

- 题库在开发和测试阶段执行结构校验；无效题目被报告并阻止发布构建通过。
- `localStorage`、Canvas、Web Speech、Web Share 和 Service Worker 均做能力检测与降级。
- 图片加载失败时使用品牌化占位图，不阻断答题。
- 分享 API 不可用时回退到复制文本或下载海报。
- Service Worker 仅在生产环境注册，缓存版本随发布更新，避免开发时缓存旧页面。
- 支持当前主流 Chromium、Firefox 和 Safari；触控目标不小于 44 px，键盘焦点始终可见。

## 8. 测试与质量门禁

### 自动测试

- 题库：ID 唯一、选项与答案合法、分类和难度有效、图片引用存在。
- 抽题：题量正确、无重复、分类覆盖满足规则、难度分布在允许范围内。
- 计分：正确/错误、速度奖励、连击、百分比和上限边界。
- 角色匹配：输入相同时结果稳定，分数范围合法，平局处理可预测。
- 存储：空数据、损坏 JSON、旧版本迁移和重置。

### 发布检查

- `npm run lint`
- `npm run test`
- `npm run build`
- 关键桌面端与移动端流程浏览器冒烟测试
- 构建产物链接、PWA manifest 和资源路径检查

GitHub Actions 在每次推送和拉取请求上执行质量门禁；`main` 分支通过后自动发布 Pages。

## 9. GitHub 包装

README 采用作品集叙事顺序：封面、在线体验、项目亮点、截图、技术架构、快速开始、脚本、目录、设计决策、测试与部署、版权说明。仓库同时提供：

- 清晰的描述、主题标签和 Pages 地址。
- MIT `LICENSE`，并明确排除第三方图片和相关作品商标。
- `THIRD_PARTY_ASSETS.md` 汇总来源、归属和非商业展示用途。
- `.gitignore` 排除依赖、构建产物、本地视觉协作文件和压缩包。
- 可复现的 `npm ci`、`npm run dev`、`npm run test` 和 `npm run build` 流程。

## 10. 非目标

- 不新增账号、后端、云数据库或排行榜服务。
- 不将娱乐测试包装成科学或心理测评。
- 不重写全部题目、角色和成就内容。
- 不宣称拥有第三方动漫、游戏图片或商标版权。
- 不引入大型 UI 框架或状态管理库。

## 11. 验收标准

- 新用户能按 README 在本地成功安装、运行、测试和构建。
- GitHub Actions 质量检查通过，GitHub Pages 可从公开地址访问。
- 首页、模式选择、完整答题、结果、历史、成就与分享核心流程正常。
- 桌面端和 375 px 宽手机端无横向溢出，主要交互可用键盘完成。
- 仓库不包含密钥、`node_modules`、构建产物、过期压缩包或视觉协作临时文件。
- 第三方素材的来源、版权和许可证边界在仓库首页可见。
